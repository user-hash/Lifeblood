# MCP Protocol Invariants

JSON-RPC 2.0 / MCP wire-format contracts, tool-registry shape, and the
truth envelope that ships on every read-side response. The protocol
contract is owned by `McpDispatcher` in `Lifeblood.Server.Mcp`;
`Program.cs` is a thin stdio I/O loop that delegates every request to
the dispatcher. It contains no routing logic, no method-string matching,
no response construction. All of that lives in `McpDispatcher`, which is
a public sealed class testable via its normal public API.

## Wire Protocol

- **INV-MCP-001. `initialize` response carries `protocolVersion` and `capabilities`.** Both are required by the MCP spec. `protocolVersion` is sourced from `McpProtocolSpec.SupportedVersion` via `McpDispatcher.SupportedProtocolVersion` (see INV-MCP-003). `capabilities` is a typed `McpCapabilities` POCO, never an inline dictionary, so new MCP capability negotiations are structural additions. Class-level defaults on `McpInitializeResult` plus explicit construction-site assignment make this belt-and-braces. Pinned by `McpProtocolTests.Initialize_ReturnsSpecCompliantResult`, `Initialize_SerializedJson_HasProtocolVersionAndCapabilities`, and `McpProtocolSourceOfTruthTests.McpDispatcher_InitializeResponse_AdvertisesSpecVersion`.

- **INV-MCP-002. Notifications never receive a response body.** Messages with `request.Id == null` are notifications per JSON-RPC 2.0. Recognized notification method names live in `McpProtocolSpec.AllKnownNotifications` (see INV-MCP-003). `McpDispatcher.KnownNotifications` is constructed directly from that set, so the two can never drift. Today's set: `notifications/initialized` (spec canonical), `initialized` (legacy alias during deprecation), `notifications/cancelled`, `$/cancelRequest`. Any entry short-circuits `Dispatch` to return `null` before response construction. Unknown notifications also return `null` and log to stderr for operator visibility. Pinned by `McpProtocolTests.NotificationsInitialized_SpecCompliantForm_ProducesNoResponse`, `NotificationsInitialized_LegacyAlias_ProducesNoResponse`, `UnknownNotification_ProducesNoResponse`, and the spec-level `McpProtocolSourceOfTruthTests.McpDispatcher_KnownNotification_ProducesNoResponse` Theory.

- **INV-MCP-003. Every MCP wire-format constant has exactly one canonical source per side.** Protocol version, JSON-RPC method names, and notification method names live exclusively in `Lifeblood.Connectors.Mcp.McpProtocolSpec`. Clients that cannot take a project reference (Unity bridge today) ship a standalone mirror file compared byte-equal to the spec by ratchet test. Unity mirror: `unity/Editor/LifebloodBridge/McpProtocolConstants.cs`, pinned by `McpProtocolSourceOfTruthTests`. Server consumes the spec directly via project reference: `McpDispatcher.SupportedProtocolVersion` chains to `McpProtocolSpec.SupportedVersion`, `McpDispatcher.KnownNotifications` builds from `McpProtocolSpec.AllKnownNotifications`, the dispatcher's method switch uses `McpProtocolSpec.Methods.*` const cases. Adding a new wire constant: edit `McpProtocolSpec` once; the mirror and consumer ratchets catch every forgotten downstream.

- **INV-MCP-STRICT-JSON-001. Strict JSON mode rejects duplicate object properties before request binding, behind an explicit operator opt-in.** Default mode preserves System.Text.Json compatibility: if a client sends duplicate properties, the normal deserializer behavior applies. `LIFEBLOOD_JSON_COMPAT=strict` is the primary operator switch; the older `LIFEBLOOD_STRICT_JSON` truthy flag remains a backward-compatible strict alias when `LIFEBLOOD_JSON_COMPAT` is unset. In strict mode `Program.cs` routes each stdio frame through `McpJsonRequestParser.DeserializeRequest(..., strictJson: true)` before dispatch. The parser walks the raw UTF-8 JSON with `Utf8JsonReader` and throws `JsonException` on the first duplicate property name in any object, including nested `params` / `arguments` objects. The existing Program parse-error path turns that into JSON-RPC `-32700` with `id: null`, so malformed frames fail at the protocol boundary rather than reaching tool handlers with ambiguous arguments. `LIFEBLOOD_JSON_COMPAT=legacy|warn|strict` is parsed by `ToolJsonCompatibilityModeReader`; unknown values fall back to legacy for deployment safety. Pinned by `McpJsonRequestParserTests` and `ToolArgumentContractTests`.

- **INV-MCP-TRANSPORT-RESILIENCE-001. No single request fault may close the stdio transport for every other call.** The read-dispatch-write loop lives in `McpServerLoop.RunAsync` (extracted from `Program.Main` so it is unit-testable through its public API, same no-visibility-tricks ethos as `McpDispatcher`). The loop is **single-flight by construction** — read one line, dispatch synchronously, write one response, then read the next — so an MCP client's parallel tool batch is serialized server-side and two compilation requests never run concurrently (the apparent "concurrent compile-check" in `LB-TRACK-20260530-029` was client-side multiplexing over a pipe that had already died, not a server-side race). Three fault classes are contained rather than fatal: (1) a dispatch fault is caught and turned into an id-correlated `-32603` response carrying a structured `data` envelope (`phase` / `method` / `exceptionType` / `recoverable` / `recovery`) — the prior design built error responses with `id: null`, so a client awaiting a request by id never saw a correlated response and treated the channel as broken; (2) a response that fails to serialize still emits a minimal id-correlated error instead of silence; (3) a write to a broken output pipe is logged and swallowed — the prior design let a broken-pipe `IOException` thrown inside the catch handler's own `Console.WriteLine` escape the loop and terminate the process, closing the transport permanently. A genuinely-closed stdin still ends the loop cleanly via the null-line sentinel. Pinned by `McpServerLoopTests` (normal round-trip echoes id; dispatch fault echoes id + loop keeps serving; broken output pipe never terminates the loop + both requests dispatched; malformed JSON emits `-32700` + loop continues).

## Tool Registry

- **INV-TOOLREG-001. Tool availability dispatch is by typed enum, never by name prefix. Internal registry records and wire-format DTOs are separate types.** Every `ToolDefinition` (internal registry record in `Lifeblood.Server.Mcp`) declares `required ToolAvailability Availability { get; init; }`; omitting it is a compile error. `ToolRegistry.GetDefinitions()` returns `ToolDefinition[]` for internal consumers; `ToolRegistry.GetTools(hasCompilationState)` projects those into `McpToolInfo[]`, the wire-format DTO used in `tools/list`, applying `[Unavailable...]` description decoration when no compilation state is loaded. The wire DTO has no `Availability` field — no `[JsonIgnore]` workaround needed; System.Text.Json serializes cleanly. The split exists because conflating wire DTOs with internal records caused both a .NET 8 `[JsonIgnore]`-on-`required init` serialization bug AND a silent tool-classification miss (prefix guard didn't match `lifeblood_resolve_short_name`). Pinned by `McpProtocolTests.ToolRegistry_EveryToolHasExplicitAvailability`, `WriteSideTools_MarkedUnavailable_WhenNoCompilationState`, `ReadSideTools_NeverMarkedUnavailable`, and `ToolRegistry_ToolsList_SerializesWithoutError`.

- **INV-MCP-TOOL-ARG-CONTRACT-001. Tool argument validation lives at the MCP server edge, generated from the registered tool input schema, and never leaks MCP JSON names into Domain or Application.** `ToolInputContract.FromSchema` projects the existing `ToolDefinition.InputSchema` into typed `ToolInputContract` / `ToolArgumentContract` metadata (argument name, JSON type, required flag, array item type, description) at the handler/composition edge, keeping protocol DTOs out of the contract layer. `ToolArgumentBinder` validates the incoming `JsonElement` before handler dispatch. `legacy` mode accepts current client behavior without diagnostics; `warn` mode accepts but records `lifeblood.tool.arguments` telemetry with unknown / missing / type-mismatch / duplicate counts; `strict` mode rejects invalid arguments with an MCP tool error whose JSON body carries `failure:"tool-argument-contract"` and structured diagnostics. This is a server-host contract layer: handlers still own semantic defaults and mutual-exclusion rules, while Domain/Application remain protocol-neutral. Pinned by `ToolArgumentContractTests` and `ToolHandlerTelemetryTests`.

- **INV-MCP-SESSION-GATE-001. Retained MCP session mutation is serialized at the server-host edge, not inside Domain or Application.** `GraphSessionGate` is an `ISessionGate` backed by `ReaderWriterLockSlim`. `ToolHandler.Handle` routes session-replacing or stale-refreshing calls (`lifeblood_analyze`, `lifeblood_compile_check`) through the write gate and ordinary read-side calls through the read gate. The gate deliberately lives in `Lifeblood.Server.Mcp`: `WorkspaceSession` stays the mutable retained-session object and lower layers stay free of daemon/threading policy. The current stdio loop is still single-flight by transport design; this gate is the narrow future-shared-server policy seam and the regression surface for analyze replacement during reads. Pinned by `GraphSessionGateTests` and `ToolHandlerTelemetryTests`.

## Wire Contract

- **INV-WIRE-CONTRACT-001. Every publicly observable wire contract (response envelope, MCP tool input + output shapes, graph JSON schema, canonical id format, parity diagnostic ID set) is pinned by a snapshot ratchet and evolved through the documented deprecation policy.** Pre-policy, Lifeblood's response shapes were tightly coupled to test code that asserted "the response has field X" but had no single ratchet preventing a quiet rename or removal. The first concrete pin lives at `ResponseEnvelopeWireShapeContractTests`: reflects on `Lifeblood.Domain.Results.ResponseEnvelope`, asserts the v1 field set is exactly six fields with their canonical names and stable type names (TruthTier / Confidence / EvidenceSource / StalenessSeconds / FilesChangedSinceAnalyze / Limitations), and additionally pins every field as `init`-only so mutation after construction is structurally blocked. MCP tool input schemas are pinned by `ToolSchemaSnapshotTests` against versioned files under `schemas/tools/v1/<tool>.schema.json`; the ratchet fails if `ToolRegistry.GetDefinitions()` drifts from the snapshots, if a registered tool has no snapshot, or if the directory keeps a stale snapshot for a removed/renamed tool. The deprecation policy (`docs/SCHEMA_DEPRECATION_POLICY.md`) names the rules: additive changes are allowed without a version bump but require updating the snapshot; renames, removals, type changes, tool removals, enum-member removals are breaking and require shipping v2 alongside v1 with at least one minor release of overlap during which v1 responses carry a `Limitations[]` deprecation notice. NEVER bypass a snapshot drift by editing the expected set without bumping the schema version first; the ratchet exists exactly to prevent that pattern.

## Response Envelope

- **INV-ANALYZE-SKIPPED-PROMINENCE-001. Staleness signals get promoted into envelope `Limitations[]` entries whenever they exceed configured thresholds.** Pre-fix, `ResponseEnvelope.StalenessSeconds` and `FilesChangedSinceAnalyze` were always on the wire but consumers had to compare them against unwritten thresholds to know whether to re-analyze. The seconds + files-changed numbers stayed silent at any magnitude; a 30-day-stale graph emitted the same envelope as a fresh one. The fix routes the always-computed signal through a configurable `StalenessPolicy` (`Application/Ports/Right/IResponseDecorator.cs`) whose two thresholds — `StalenessSecondsWarnThreshold` (default 3600 s = 1 hour), `FilesChangedWarnThreshold` (default 10) — gate explicit `Limitations[]` entries. The composition root (`Server.Mcp.Program`) reads `LIFEBLOOD_STALENESS_SECONDS_THRESHOLD` / `LIFEBLOOD_FILES_CHANGED_THRESHOLD` from the environment at startup so operators tune per-deployment without code change; missing or malformed values fall back to `StalenessPolicy.Default`. The decorator appends staleness limitations AFTER the static per-tool limitations so a consumer reading from index 0 still sees the tool-specific caveats; setting a threshold to `int.MaxValue` / `long.MaxValue` effectively disables the corresponding entry. The unregistered-tool path also honors the policy — missing tool classification does not exempt the workspace from the staleness contract. Pinned by `StalenessPolicyEnvelopeTests` (5 facts: fresh graph emits no extra limitation, staleness-seconds threshold appends entry, files-changed threshold appends entry, custom thresholds override defaults, unregistered tool still surfaces staleness limitation).

- **INV-ADVISORY-LIMITATIONS-001. Every tool — read OR write side — declares an `EnvelopeClassification` in the registry, and every write-side response is wrapped through `WrapWriteSide` so it carries the same `envelope` field as read-side responses.** Pre-S6, `lifeblood_enum_coverage`, `lifeblood_static_tables`, `lifeblood_find_references`, `lifeblood_find_definition`, `lifeblood_find_implementations`, `lifeblood_symbol_at_position`, `lifeblood_documentation`, `lifeblood_diagnose`, `lifeblood_compile_check`, `lifeblood_rename`, `lifeblood_format`, `lifeblood_execute` returned raw JSON with NO envelope on the wire — callers could not distinguish a fresh from a 30-day-stale `find_references` hit list, and tools with documented Unity-runtime blind spots emitted no warning. S6 wires three new classification shorthands surfacing the limitation honestly: `DerivedProvenWithUnityRuntimeRisk` (port_health, authority_report — graph-walk is exact but Unity runtime dispatch invisible to static analysis), `SemanticProvenWithUnityRuntimeRisk` (enum_coverage — serialized-enum fields on prefabs/scenes/SOs can produce values invisibly), `DerivedProvenWithTestDiscoveryRisk` (test_impact — reflection-driven NUnit `[TestCaseSource]`, `[Theory]`, custom test attributes invisible to graph). `WrapWriteSide(toolName, McpToolResult)` in `ToolHandler` is the single seam wrapping every write-side dispatch site through `MergeEnvelopeIntoJson` so envelope shape is uniform across read + write tools. Idempotent: skips error results, skips non-JSON-object payloads, skips already-wrapped responses. The strengthened ratchet `Registry_EveryToolWithEnvelopeWrap_DeclaresEnvelopeClassification` fails when any tool is missing a classification — read or write — so a write-side tool cannot regress into the degraded "Unregistered tool" fallback without the build noticing. LB-TRACK-20260519-026 / S6 atom of the 2026-05-19 plan.

- **INV-DIAGNOSE-FRESHNESS-001. Every read-side response carries a monotonic `analysisGeneration` counter that bumps once per `WorkspaceSession.Load`.** Pre-S5, two reads taken either side of an auto-refresh (compile_check triggering `MaybeRefreshIfStale`) were indistinguishable on the wire — a cross-tool join joining a `find_references` hit list with a `compile_check` diagnostic had no way to detect that the underlying workspace state changed between them. The counter lives on `WorkspaceSession.AnalysisGeneration` (long, increments inside `Load`, survives `Clear` so two reads either side of a Clear/Load pair see distinct values), flows through `GraphSession.AnalysisGeneration` into `EnvelopeContext.AnalysisGeneration`, and lands on `ResponseEnvelope.AnalysisGeneration` via `LifebloodResponseDecorator`. Zero before the first Load. The unregistered-tool fallback in the decorator ALSO copies the generation — feature-on-fallback parity caught during S5 self-review against the F3-series mistake class (happy path wired, degraded path forgotten). Pinned by `ResponseEnvelopeTests` (6 S5 facts: fresh session zero, increments-per-load, monotonic-across-clear-load-pairs, decorator-copies-from-context, unregistered-tool-still-copies, no-workspace-zero). LB-TRACK-20260519-025 / S5 atom of the 2026-05-19 plan.

- **INV-ENVELOPE-001. Every read-side MCP tool response carries a top-level `envelope` field.** The envelope (`Lifeblood.Domain.Results.ResponseEnvelope`) ships truth-tier (Semantic / Derived / Heuristic / Inferred), confidence band (Proven / Advisory / Speculative), evidence-source string, wall-clock staleness in seconds, files-changed-since-analyze count, a per-tool limitations array, and an `analysisGeneration` monotonic counter (INV-DIAGNOSE-FRESHNESS-001). Population is owned by `IResponseDecorator` (Application port, `LifebloodResponseDecorator` reference adapter); every read-side handler in `ToolHandler` routes through `WithEnvelope(toolName, payload)` (or `MergeEnvelopeIntoJson` for the analyze response) so missing the envelope on a new tool is structurally impossible without bypassing the helper. The decorator's classification table is the single source of truth for per-tool tier/confidence; an unregistered tool falls through to the most-conservative (`Heuristic` + `Speculative`) envelope plus a `Limitations` entry naming the gap, so a missed registration surfaces as an obviously-degraded envelope rather than silent over-confidence. Pinned by `ResponseEnvelopeTests.Decorator_AllReadSideToolsInRegistry_HaveClassification`, which iterates `ToolRegistry.GetDefinitions()` and asserts every read-side tool has a real classification, plus end-to-end tests that read the wire-shape `envelope` field out of `lifeblood_lookup` / `lifeblood_blast_radius` / `lifeblood_resolve_short_name` responses. Closes LB-INBOX-001 + LB-OBS-004.

- **INV-NATIVE-CLANG-ENVELOPE-001. Imported graph envelopes are bounded by adapter capability.** Per-tool envelope classification remains the starting point, but `EnvelopeContext.AdapterCapability` carries the loaded graph's declared adapter capability into `LifebloodResponseDecorator`. Fully proven Roslyn capabilities leave existing envelopes unchanged. External / non-Roslyn graphs append a limitation naming adapter, language, version, and capability fields; if core symbol or call facts are not Proven, a Proven tool classification is downgraded to Advisory. Adapter-less legacy JSON imports are normalized at the MCP session boundary to an explicit `unknown-json-graph` capability so absence of metadata cannot mean "trust like Roslyn." This prevents a native, Python, TypeScript, or other JSON graph from inheriting Roslyn-shaped authority by accident. Pinned by `ResponseEnvelopeTests.Decorator_NativeClangCapability_AddsAdapterLimitsWithoutDowngradingProvenDirectFacts`, `Decorator_BestEffortAdapter_DowngradesProvenToolClassificationToAdvisory`, `ToolHandler_ImportedNativeGraph_EnvelopeCarriesAdapterCapability`, and `ToolHandler_ImportedGraphWithoutAdapterMetadata_GetsUnknownAdapterCapability`.
