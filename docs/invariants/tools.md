## Dead-Code Analysis (v0.6.3, major fix v0.6.4)

- **INV-DEADCODE-001. `lifeblood_dead_code` walks the graph for symbols with zero incoming non-Contains edges. Every response carries `status` + `warning` fields.** The analyzer checks `graph.GetIncomingEdgeIndexes(sym.Id)` and also checks outgoing `Implements` edges as proof of liveness (a method implementing an interface is reachable through the interface). Self-analysis tail after the post-v0.6.4 extractor pass: ~7 findings (from 150 pre-v0.6.4), all in the known-structural-limitation set below.

  **Closed false-positive classes** (v0.6.4 + post-v0.6.4 extractor pass). Each maps to one or more extractor changes in `RoslynEdgeExtractor`:
  - **Interface dispatch.** `ExtractInheritanceEdges` emits method-level `Implements` edges via `FindImplementationForInterfaceMember` + `AllInterfaces`. Dead-code analyzer checks outgoing `Implements` as proof of liveness.
  - **Member access granularity.** `ExtractMemberAccessEdge` + `ExtractReferenceEdge` emit both type-level and symbol-level `References` edges for properties/fields/method-groups via `EmitSymbolLevelEdge` shared helper. Covers bare field identifiers and method-group references (`new Lazy<>(Load)`, `event += Handler`).
  - **Null-conditional property access.** `MemberBindingExpressionSyntax` handler emits type-level + symbol-level edges for `obj?.Property`.
  - **Lambda / local-function context.** `FindContainingMethodOrLocal` skips lambda and local-function syntax nodes. Calls inside `.Select(x => Foo(x))` attribute to the enclosing named method.
  - **Implicit global usings.** `ModuleInfo.ImplicitUsings` discovered from csproj; `ModuleCompilationBuilder.CreateCompilation` injects a synthetic global-using tree when enabled. Closes the 42% `GetSymbolInfo` null-resolution class (without this, `List<>`, `Dictionary<>` etc. emit CS0246 on every module). Follows INV-COMPFACT pattern.
  - **Constructor `Calls` edge.** `ExtractConstructorCallEdge` emits BOTH a type-level `References` edge (module-coupling signal) AND a method-level `Calls` edge to the `.ctor`. `find_references` on any constructor returns its construction sites.
  - **Field-initializer containing method.** `FindContainingMethodOrLocal` resolves a reference inside `static T _x = Bar()` / `T _x = Bar()` to the type's synthesized `.cctor` / first `.ctor` via `StaticConstructors` / `InstanceConstructors`.
  - **Property accessor context.** `FindContainingMethodOrLocal` returns the accessor `IMethodSymbol`; `GetMethodId` maps `AssociatedSymbol` to the property/event id so the emitted edge source matches the extracted graph node. Covers bodied accessors, expression-bodied properties, and indexer expression bodies.

  **Remaining known false-positive classes (structural, not fixable by static analysis):**
  - **Runtime entry points.** `Program.Main` and `Program` types in composition roots. Self-analysis: 6 findings.
  - **Unity reflection-based dispatch.** Methods called by the Unity engine at runtime via `[RuntimeInitializeOnLoadMethod]`, lifecycle messages (`OnAudioFilterRead`, `OnApplicationFocus`), and `SendMessage`-dispatched handlers. Roslyn cannot see these call sites. Full audit on a real 75-module Unity workspace: 96% true-positive rate (25/26 verified).

  Pinned by `Handle_DeadCode_Response_IncludesExperimentalWarning` plus 26 extractor/analyzer tests in `RoslynExtractorTests` covering each closed FP class.

## Invariant Introspection (v0.6.3, tree-walker added LB-FR-023)

- **INV-INVARIANT-001. Architectural invariants live across well-known repo conventions; the tool aggregates them at runtime.** `lifeblood_invariant_check` discovers source files dynamically via `IFileSystem` â€” `<root>/CLAUDE.md`, `<root>/AGENTS.md`, and any `<root>/docs/invariants/**.md` tree â€” parses each with `Internal.ClaudeMdInvariantParser`, caches per-file in `InvariantParseCache<T>` (timestamp-based invalidation), and aggregates results across all sources. Five authoring shapes recognised: A (`- **INV-X-N**: body`), B (`- **INV-X-N. Title.** Body`), C (`**INV-X-N: Title.** body` â€” bare bold paragraph, no bullet), D (`- **INV-X-N** (vX.Y.Z): body` â€” parenthesized version tag), E (`- **INV-X-N:** body` â€” colon inside the bold). Three modes via `IInvariantProvider`: `id` lookup (full body), `audit` (total + per-category breakdown + duplicate ids + parse warnings + every contributing source path in `SourcePaths[]`), `list` (id+title index, no bodies). Parser is a pure function; provider is a thin orchestrator over discovery + cache + parser â€” adding a new invariant source layout (YAML companion file, GitHub issue tracker, external governance DB) ships as a sibling provider reusing the cache. Duplicate ids declared across files are detected and surfaced. Pinned by `ClaudeMdInvariantParserTests`, `InvariantProviderAndHandlerTests`, `LifebloodClaudeMdSelfTests`, `InvariantParseCacheTests`. Closes LB-BUG-017 / LB-BUG-018 / LB-FR-023.

## Authority Report + Forwarder Classifier (v0.6.7 P5)

- **INV-AUTHORITY-001. Authority report quantifies type-level architectural responsibility.** `IAuthorityReporter.Analyze(graph, typeId)` returns implementedInterfaceCount (distinct outgoing `Implements` edges to interface types), ownedPublicSurface (public method/property/field count from `Contains` edges, nested types excluded), per-implemented-interface usage (member count + distinct consumers reaching the interface or any of its members via `Calls`/`References`), and a forwarderRatio (`PureForwarderCount / TotalMethodCount` in [0.0, 1.0]; sentinel `-1.0` when no method has the extractor-recorded `classification` property). Single graph walk produces every field. The reporter is stateless (INV-ANALYSIS-001), reads only the graph, and lives in `Connectors.Mcp` because `Lifeblood.Analysis` cannot reference Application ports (INV-CONN-001). Surfaced as the `lifeblood_authority_report` MCP tool. **Use cases are general â€” not extraction-specific.** Any type that aggregates surface across multiple subordinates is in scope: partial-class hosts, dispatchers that route across many row paths (filter family selectors, kernel routers, capability dispatchers), facade types fronting an internal subsystem, ports with many implementations. The DAWG dogfood case happened to be an ABG partial-class wave; the same metrics drive every other host-with-many-subordinates triage. Pinned by `AuthorityReporterTests.AuthorityReport_*`.

- **INV-FORWARDER-001. Method body shape is classified at extraction time, recorded on `Properties["classification"]`.** Three categories: `PureForwarder` (single-statement / expression-bodied method whose body is exactly one invocation expression), `ThinWrapper` (â‰¤ 5 statements containing exactly one invocation overall â€” null-guard-then-delegate, simple-cast-then-delegate), `RealLogic` (anything else). Abstract / partial / extern methods are not classified (no entry written) â€” there's no body shape to categorize. Heuristic, not semantic â€” recorded on the type's own member, not the call target â€” but stable enough that an authority report can compute a coarse forwarder ratio without a second walk. Detected by `RoslynSymbolExtractor.AttachMethodClassification` (Phase P5 / LB-FR-020). Pinned by `AuthorityReporterTests.Extractor_*`.

## Execute Robustness (v0.6.7 P4)

- **INV-EXECUTE-001. `lifeblood_execute` references three reference sets, in this order, every call.** (1) The selected target-profile BCL â€” host runtime by default, or a specific reference pack (`net-standard-2.1`, `net-6.0`) when the caller passes `targetProfile`; missing ref-packs fall back to the host BCL with a `targetRuntimeWarnings` entry instead of a hard failure. (2) Every retained workspace `CSharpCompilation` as a `MetadataReference`. (3) When the workspace is Unity-shaped (`Library/` exists at the project root), the `IRuntimeAssemblyResolver` adapter (`Lifeblood.Adapters.CSharp.UnityAssemblyResolver`) injects DLLs from `Library/ScriptAssemblies/`, `Library/Bee/artifacts/`, and `Library/PackageCache/` so scripts can touch `UnityEngine` types; an empty `Library/` surfaces a `runtimeAssemblyWarnings` entry telling the caller to run a Unity build first. The script-globals object (`RoslynSemanticView`) ships `Help`, `SymbolsOfKind(string)`, and `EdgesOfKind(string)` so scripts can introspect the sandbox without importing the `SymbolKind` / `EdgeKind` enums. Pinned by `ExecuteRobustnessTests` (13 tests) â€” resolver no-Library / library-without-artifacts / DLL discovery; target-profile host / unknown-fallback / legacy-overload delegation; warnings propagation through both success and blocked-pattern paths; `Help` content shape; string-named `SymbolsOfKind` / `EdgesOfKind` filtering. Closes LB-BUG-014, LB-BUG-007, LB-FR-012, LB-FR-013, LB-BUG-008.

## Unity Reachability (v0.6.7 P3)

- **INV-UNITY-001. Runtime-dispatch reachability is a typed port. `IDeadCodeAnalyzer` consults `IUnityReachabilityProvider` (when injected) before flagging a symbol as dead.** The reference adapter (`Lifeblood.Adapters.CSharp.UnityReachabilityAdapter`) recognizes Unity entrypoint attributes (`RuntimeInitializeOnLoadMethod`, `InitializeOnLoadMethod`, `MenuItem`, `ContextMenu`, `PostProcessBuild`, `PostProcessScene`, `CustomEditor`, `CustomPropertyDrawer`, `Test`, `UnityTest`, â€¦) and MonoBehaviour magic methods on types whose inheritance chain reaches a Unity message-receiver base (`UnityEngine.MonoBehaviour`, `UnityEngine.ScriptableObject`, `UnityEditor.Editor`, `UnityEditor.EditorWindow`, `UnityEngine.StateMachineBehaviour`). The adapter walks the chain via the extractor-recorded `Symbol.Properties["baseType"]` (FQN) so chains terminating in external types not loaded into the graph (UnityEngine.dll lives outside the workspace) still resolve. Cycle-safe with a 32-hop cap. Composition root injects the adapter by default; non-Unity workspaces see zero behavior change because nothing matches. Dogfooded on a real 87-module Unity workspace: dead_code findings 1095 â†’ 729 (-33%), MonoBehaviour-magic FPs 378 â†’ 13 (-97%). Pinned by `UnityReachabilityTests` (15 tests) covering attribute matching, magic-method detection on direct + transitive subclasses (in-graph and via external base FQN), negative cases, cycle-safety, extractor `Properties["baseType"]` + `Properties["attributes"]` recording, and end-to-end through `LifebloodDeadCodeAnalyzer`. Closes LB-FP-001, LB-FP-002, LB-BUG-009.

- **INV-UNITY-002. Asmdef edits force a full incremental re-analyze.** `AnalysisSnapshot.AsmdefTimestamps` records `*.asmdef` mtime at full-analyze time. `RoslynWorkspaceAnalyzer.IncrementalAnalyze.HasAsmdefDrift` re-scans the project root on every incremental call; any added, edited, or removed `.asmdef` triggers a full re-analyze that round. Necessary because Unity csprojs are generated from asmdefs â€” editing an asmdef without forcing Unity to regenerate the csproj leaves the csproj-timestamp tracker (INV-BCL-005) blind to the change, which silently runs analysis against stale module facts. Pinned by `UnityReachabilityTests` end-to-end coverage; closes the silent-stale-module class promoted from LB-NICE-003.

## Semantic Search Match Source

- **INV-SEARCH-MATCHKIND-001. `SearchResult.MatchKind` structurally reports which scoring bucket(s) drove the rank.** The reference adapter (`Lifeblood.Connectors.Mcp.LifebloodSemanticSearchProvider`) ranks symbols against three buckets â€” bare `Name`, `QualifiedName` (exclusive of Name per the bare-name guard), and persisted `xmlDocSummary`. Pre-fix the bucket signal lived only in the human-rendered `MatchSnippets` strings (`"name: â€¦"` / `"qualifiedName: â€¦"` / `"xmlDoc: â€¦"`); callers who wanted to filter by source had to parse those strings. The first-hit-token trackers inside the adapter already knew which buckets fired â€” INV-SEARCH-MATCHKIND-001 lifts that to a typed `MatchKind` enum on `SearchResult`. Values: `Name`, `QualifiedName`, `XmlDoc` for single-bucket hits; `Multiple` when more than one bucket fired (e.g. token in both bare name and xmldoc summary, or qualified name and xmldoc summary). The enum is adapter-agnostic â€” future TypeScript / Python search adapters reuse the same taxonomy (their corpora carry the same conceptual buckets). Snippet strings are unchanged so existing renderers keep working; the structured field is purely additive. Pinned by `SemanticSearchTests.Search_*MatchKind*` (5 cases: name-only, qualifiedName-only-with-bare-name-guard, xmldoc-only, name+xmldoc Multiple, qualifiedName+xmldoc Multiple).

## Find-Implementations Comparison (v0.6.1)

- **INV-FINDIMPL-001. `FindImplementations` compares via canonical Lifeblood symbol IDs, never display strings or `SymbolEqualityComparer`.** The canonical ID from `RoslynCompilationHost.BuildSymbolId` (routing through `CanonicalSymbolFormat`) is the only cross-assembly-safe comparison for C# symbols. `ToDisplayString()` is subject to the v0.6.0 nullability/reduced-name/attribute drift class. `SymbolEqualityComparer.Default` is stricter than needed â€” treats source and PE-downgraded metadata copies as unequal because they live in different assemblies, breaking the cross-module matching `FindImplementations` needs. This invariant generalizes to ALL write-side matching across the source/metadata boundary: always route through `BuildSymbolId`. Pinned by `WriteSideIntegrationTests.FindImplementations_IGreeter_FindsGreeterAndFormalGreeter`.
