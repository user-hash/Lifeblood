## Dead-Code Analysis (v0.6.3, major fix v0.6.4)

- **INV-DEADCODE-001. `lifeblood_dead_code` walks the graph for symbols with zero incoming non-Contains edges. Every response carries `status` + `warning` fields.** The analyzer checks `graph.GetIncomingEdgeIndexes(sym.Id)` and also checks outgoing `Implements` edges as proof of liveness (a method implementing an interface is reachable through the interface). Self-analysis tail after the post-v0.6.4 extractor pass: ~7 findings (from 150 pre-v0.6.4), all in the known-structural-limitation set below.

  **Closed false-positive classes** (v0.6.4 + post-v0.6.4 extractor pass). Each maps to one or more extractor changes in `RoslynEdgeExtractor`:
  - **Interface dispatch.** `ExtractInheritanceEdges` emits method-level `Implements` edges via `FindImplementationForInterfaceMember` + `AllInterfaces`. Dead-code analyzer checks outgoing `Implements` as proof of liveness.
  - **Member access granularity.** `ExtractMemberAccessEdge` + `ExtractReferenceEdge` emit both type-level and symbol-level `References` edges for properties/fields/method-groups via `EmitSymbolLevelEdge` shared helper. Covers bare field identifiers and method-group references (`new Lazy<>(Load)`, `event += Handler`).
  - **Null-conditional property access.** `MemberBindingExpressionSyntax` handler emits type-level + symbol-level edges for `obj?.Property`.
  - **Lambda / local-function context.** `FindContainingMethodOrLocal` skips lambda and local-function syntax nodes. Calls inside `.Select(x => Foo(x))` attribute to the enclosing named method.
  - **Implicit global usings.** `ModuleInfo.ImplicitUsings` discovered from csproj; `ModuleCompilationBuilder.CreateCompilation` injects a synthetic global-using tree when enabled. Closes the 42% `GetSymbolInfo` null-resolution class (without this, `List<>`, `Dictionary<>` etc. emit CS0246 on every module). Follows INV-COMPFACT pattern.
  - **Constructor `Calls` edge.** `ExtractConstructorCallEdge` emits BOTH a type-level `References` edge (module-coupling signal) AND a method-level `Calls` edge to the `.ctor`. `find_references` on any constructor returns its construction sites.
  - **Field-initializer containing method.** `FindContainingMethodOrLocal` resolves a reference inside `static T _x = Bar()` / `T _x = Bar()` to the type's synthesized `.cctor` / first `.ctor` via `StaticConstructors` / `InstanceConstructors`.
  - **Property accessor context.** `FindContainingMethodOrLocal` returns the accessor `IMethodSymbol`; `GetMethodId` maps `AssociatedSymbol` to the property/event id so the emitted edge source matches the extracted graph node. Covers bodied accessors, expression-bodied properties, and indexer expression bodies.

  **Remaining known false-positive classes (structural, not fixable by static analysis):**
  - **Runtime entry points.** `Program.Main` and `Program` types in composition roots. Self-analysis: 6 findings.
  - **Unity reflection-based dispatch.** Methods called by the Unity engine at runtime via `[RuntimeInitializeOnLoadMethod]`, lifecycle messages (`OnAudioFilterRead`, `OnApplicationFocus`), and `SendMessage`-dispatched handlers. Roslyn cannot see these call sites. Full audit on a real 75-module Unity workspace: 96% true-positive rate (25/26 verified).

  Pinned by `Handle_DeadCode_Response_IncludesExperimentalWarning` plus 26 extractor/analyzer tests in `RoslynExtractorTests` covering each closed FP class.

- **INV-DEADCODE-TRIAGE-003. `lifeblood_dead_code` accepts `pathExclude` glob patterns that drop findings by file path.** Additive to INV-DEADCODE-TRIAGE-001/002. `DeadCodeOptions.PathExclude` is an optional set of globs matched against each symbol's normalized POSIX file path inside `LifebloodDeadCodeAnalyzer.FindDeadCode`; any match excludes the symbol before liveness analysis. Glob grammar: `*` = any run of characters including `/`, `?` = a single character, every other character escaped literally; matching is case-insensitive and anchored (FULL-path match, so substring intent needs leading/trailing `*`). Null / empty patterns (and whitespace-only entries) are no-ops, so the default response is unchanged. Purpose: fold vendored / sample / third-party roots (`*/Examples*/*`, `*/Samples*/*`, `Packages/*`) out of dead-code triage without a breaking change to the `bucket` enum — the heavier `Vendored` bucket reclassification stays deferred to its own atom. Pinned by `DeadCodeTriageFieldsTests.FindDeadCode_PathExclude_DropsVendoredFindingsKeepsProduction` + `FindDeadCode_PathExclude_NullOrEmpty_ChangesNothing`. First half of `LB-INTAKE-20260601-004`.

- **INV-DEADCODE-TRIAGE-002. Dead-code findings carry `sameClassConsumerCount`; `DeadCodeOptions.IncludeSameClassOnlyConsumers` opt-in surfaces symbols whose only consumers share the candidate's containing type.** Additive to INV-DEADCODE-TRIAGE-001. Default mode keeps the strict liveness filter — any symbol with an incoming non-Contains edge is excluded — so the count is `0` for every classic finding. When `IncludeSameClassOnlyConsumers=true`, the analyzer relaxes the filter to also surface symbols whose every incoming non-Contains reference originates from a member with the same `ParentId` (same containing type); the finding then reports the count plus a `Reason` of `"<scope><kind> with only same-class consumers (<N>)"`. Cross-class consumers (even one) suppress the relaxed surfacing — the symbol is genuinely consumed outside its declaring class and not a class-internal cleanup candidate. `IsLiveByOutgoingImplements` is split out of the legacy `HasIncomingReference` helper so the relaxation cannot accidentally surface interface implementers. Use case: class-level cleanup triage. A private field, property, or method whose only consumers are sibling members of the same class is a valid candidate for inlining or removal during a focused class-by-class pass, even though it is semantically alive. `SameClassConsumerCount` is always populated on the wire so consumers can sort/filter, even when the relaxation is off (returns `0` in classic mode). Pinned by `AnalysisToolsTests.DeadCode_DefaultMode_PrivateFieldWithSameClassConsumer_IsNotFlagged` (default filtering preserved), `DeadCode_RelaxedMode_PrivateFieldWithSameClassConsumer_IsFlaggedWithCount`, `DeadCode_RelaxedMode_PrivateFieldWithCrossClassConsumer_IsNotFlagged`, and `DeadCode_TrulyUnusedPrivateField_ReportsZeroSameClassConsumers`. LB-TRACK-20260515-015 / F2 atom of the 2026-05-19 plan.

- **INV-DEADCODE-TRIAGE-001. Every dead-code finding carries triage fields: `directDependants`, `bucket`, `declarationOnly`.** First-response triage shape for a tool whose raw findings list overwhelms agents on real workspaces. `directDependants` is the count of incoming non-Contains edges (always `0` for classic findings the analyzer emits, but on the wire as forward-compatible signal for future relaxed-criteria modes — reachable-only-via-Implements, reachable-only-via-Override — where the "barely reachable" class would surface with a non-zero count). `bucket` is the segment-aware path classification (`Production` / `Test` / `Editor` / `Generated`) mirroring the `blast_radius groupBy=bucket` taxonomy so a caller can join the two tool surfaces; classification is performed on whole path segments (not substrings) and follows precedence Generated → Test → Editor → Production (most-authoritative signal wins). Test beats Editor by design: a fixture under `Tests/Editor/Foo.cs` is a test fixture (defined by its `Tests` root plus filename convention), not an Editor utility — the `Editor/` subfolder there is just NUnit PlayMode assembly placement. Generated wins over everything because build artifacts (`obj/`, `bin/`) and codegen (`*.Generated.*`, `/generated/`) are never a refactor target regardless of any other signal. `declarationOnly` is true iff the underlying `Symbol.IsAbstract` is set — abstract methods, abstract types, interface members; deleting one is a public-contract change for every implementor and must not be treated as a routine cleanup. The response additionally carries a `bucketBreakdown` count map alongside the existing `kindBreakdown` so a caller can fold the giant Editor/Generated tail without re-walking `findings[]`. Computed once per finding at `LifebloodDeadCodeAnalyzer.FindDeadCode` time. Pinned by `DeadCodeTriageFieldsTests` (bucket-classification theory + DeclarationOnly + DirectDependants invariant). Closes LB-TRACK-004.

## Invariant Introspection (v0.6.3, tree-walker added LB-FR-023)

- **INV-INVARIANT-001. Architectural invariants live across well-known repo conventions; the tool aggregates them at runtime.** `lifeblood_invariant_check` discovers source files dynamically via `IFileSystem` — `<root>/CLAUDE.md`, `<root>/AGENTS.md`, and any `<root>/docs/invariants/**.md` tree — parses each with `Internal.ClaudeMdInvariantParser`, caches per-file in `InvariantParseCache<T>` (timestamp-based invalidation), and aggregates results across all sources. Five authoring shapes recognised: A (`- **INV-X-N**: body`), B (`- **INV-X-N. Title.** Body`), C (`**INV-X-N: Title.** body` — bare bold paragraph, no bullet), D (`- **INV-X-N** (vX.Y.Z): body` — parenthesized version tag), E (`- **INV-X-N:** body` — colon inside the bold). Three modes via `IInvariantProvider`: `id` lookup (full body), `audit` (unique `totalCount` + `declaredCount`/`duplicateDeclarationCount` reconciliation + per-category breakdown + `duplicates` with per-site `occurrences[]` provenance + parse warnings + per-source declaration counts in `sourceCounts[]` alongside `sourcePaths[]`), `list` (id+title index, no bodies). Parser is a pure function; provider is a thin orchestrator over discovery + cache + parser — adding a new invariant source layout (YAML companion file, GitHub issue tracker, external governance DB) ships as a sibling provider reusing the cache. Duplicate ids declared more than once — within a single file OR across files (e.g. a CLAUDE.md ↔ AGENTS.md mirror) — are detected and surfaced: each duplicate carries a per-site `occurrences[]` ledger (`sourcePath` + `line` + `title`) so a caller distinguishes a benign mirror (identical titles) from a real collision (divergent titles), and the audit's `declaredCount` (= Σ `sourceCounts`) minus `totalCount` (unique ids) equals `duplicateDeclarationCount`, reconciling the per-source sums against the unique total. Pinned by `ClaudeMdInvariantParserTests`, `InvariantProviderAndHandlerTests`, `LifebloodClaudeMdSelfTests`, `InvariantParseCacheTests`. Closes LB-BUG-017 / LB-BUG-018 / LB-FR-023.

The implementation is now `LifebloodPortHealthAnalyzer` in `Lifeblood.Connectors.Mcp`, behind the `IPortHealthAnalyzer` port in `Lifeblood.Application.Ports.Right`. `ToolHandler` injects the port (default constructor falls back to `new LifebloodPortHealthAnalyzer()` to match the prior `IAuthorityReporter` pattern) and the tool handler shrinks to argument parsing + result shaping. - **INV-PORT-HEALTH-COMPOSITE-001. `port_health` reports direct + inherited interface surface; liveness verdict is computed across the aggregate.** Pre-F3b, the analyzer counted only directly-declared members reached by outgoing `Contains` edges. Composite interfaces — types that inherit from one or more sub-interfaces and declare little or no surface themselves — appeared as `empty` even when their inherited contracts carried real load-bearing surface. F3b walks outgoing `EdgeKind.Inherits` edges transitively from the target type (the shared `Lifeblood.Domain.Graph.InterfaceInheritanceWalker`, also used by `authority_report` post-F3e), collecting every reachable interface's direct members (distinct by canonical id), and computes verdict + `liveMembers` + `live[]` / `dead[]` across the union of (direct ∪ inherited). Five new wire fields on `PortHealthReport`: `directMemberCount` (pre-F3b count, what this type contributes itself), `inheritedMemberCount` (distinct ids reachable through the inheritance closure), `aggregateMemberCount` (sum), `inheritedInterfaces[]` (canonical ids, sorted ordinal so the wire shape is deterministic), `isCompositeInterface` (true iff `inheritedMemberCount > 0`). `memberCount` retains its pre-F3b name as a backwards-compatible alias for `aggregateMemberCount` — older callers still get a verdict-relevant total without code change. Pinned by `PortHealthAnalyzerTests` (6 F3b fixtures: non-composite zero-inherited, composite with all-live inherited, composite mixed-live inherited, transitive grandparent walk distinct across closure, empty composite reports `empty`, composite with own direct members aggregates both surfaces). ## Authority Report + Forwarder Classifier (v0.6.7 P5, post-v0.7.7 F3e)

- **INV-AUTHORITY-001. Authority report quantifies type-level architectural responsibility.** `IAuthorityReporter.Analyze(graph, typeId)` returns implementedInterfaceCount (distinct interfaces directly satisfied — outgoing `Implements` edges for class/struct sources, outgoing `Inherits` edges for interface sources per INV-EXTRACT-IFACE-INHERIT-001), ownedPublicSurface (public method/property/field count from `Contains` edges, nested types excluded), per-implemented-interface usage (direct + inherited member counts + distinct consumers reaching the interface or any of its members via `Calls`/`References`, see INV-AUTHORITY-COMPOSITE-001), and a forwarderRatio (`PureForwarderCount / TotalMethodCount` in [0.0, 1.0]; sentinel `-1.0` when no method has the extractor-recorded `classification` property). Single graph walk produces every field. The reporter is stateless (INV-ANALYSIS-001), reads only the graph, and lives in `Connectors.Mcp` because `Lifeblood.Analysis` cannot reference Application ports (INV-CONN-001). Surfaced as the `lifeblood_authority_report` MCP tool. **Use cases are general — not extraction-specific.** Any type that aggregates surface across multiple subordinates is in scope: partial-class hosts, dispatchers that route across many row paths (filter family selectors, kernel routers, capability dispatchers), facade types fronting an internal subsystem, ports with many implementations. The original dogfood case was a partial-class wave on a real Unity workspace; the same metrics drive every other host-with-many-subordinates triage. Pinned by `AuthorityReporterTests.AuthorityReport_*`.

- **INV-AUTHORITY-PLANNING-COMPOSITION-001. Planning verdicts (EvictableDebt / BoundaryContract / SceneDiscoveryContract / CompositeFacade / AdapterShimOnly / NeedsAudit) compose CLIENT-SIDE from the evidence fields `authority_report` returns; Lifeblood ships measurements, not verdicts.** Pre-S7 a caller needed three round trips (authority_report + dependants + find_implementations) to triage whether an interface was a real boundary contract or an evictable adapter shim. S7 adds three evidence fields to the top-level `AuthorityReport`: `crossAssemblyConsumerCount` (distinct other modules with any incoming Calls/References edges into the type or its direct members), `sameAssemblyConsumerCount` (distinct same-module consumer symbols), `hasSingleImplementer` (true/false for interface targets with exactly one source-defined Implements edge; null for non-interface targets). Combined with the F3e per-interface `aggregateMemberCount` + the existing `forwarderRatio` + `ownedPublicSurface`, callers derive verdicts: **BoundaryContract** = high `crossAssemblyConsumerCount` + low `forwarderRatio` + multiple implementers (or none-yet-by-design); **AdapterShimOnly** = zero `crossAssemblyConsumerCount` + high `sameAssemblyConsumerCount` + `hasSingleImplementer == true`; **CompositeFacade** = `PerInterface[*].isCompositeInterface == true` carrying the load-bearing surface; **EvictableDebt** = `forwarderRatio > 0.6` + low `ownedPublicSurface` + concentrated same-assembly use; **SceneDiscoveryContract** = Unity workspace + zero cross-assembly use but the type carries Unity attributes (caller checks `Symbol.Properties["attributes"]`); **NeedsAudit** = no obvious shape fits, evidence is mixed. The recipe mirrors INV-FLAG-COVERAGE-COMPOSITION-001's precedent — Lifeblood is an instrument, not an opinion engine; verdict-shaped wire tools are forbidden so the caller's domain knowledge stays load-bearing. Evidence computation lives in `LifebloodAuthorityReporter.ClassifyConsumersByAssembly` + `ComputeHasSingleImplementer` + `FindContainingModule` (walks `Symbol.ParentId` up to a Module ancestor with a 16-hop safety cap). Pinned by 8 S7 fixtures in `AuthorityReporterTests`: boundary-shape (cross > 0, same = 0), adapter-shim-shape (cross = 0, same > 0), no-module-ancestor degrades to (0, 0), hasSingleImplementer true for one-impl interface, false for multi-impl, null for non-interface, null for missing typeKind, real-graph end-to-end through the production extractor. LB-TRACK-20260519-027 / S7 atom of the 2026-05-19 plan.

- **INV-AUTHORITY-COMPOSITE-001. `InterfaceUsage` rows on an authority report carry direct + inherited member surface; composite interface facades aggregate inherited contracts.** Pre-F3e, each per-interface row reported only the satisfied interface's directly-declared members (outgoing `Contains` edges from the interface itself). For ABG-style composite facades — an interface that declares little or no surface of its own but inherits 2-N sub-interfaces — this gave `MemberCount: 0` and made the row look architecturally empty even when the inherited contract carried the real load-bearing surface. F3e reuses the shared `InterfaceInheritanceWalker` (also used by `port_health`'s F3b) to walk the satisfied interface's transitive `EdgeKind.Inherits` closure and sum each parent's direct member count. Five new wire fields on `InterfaceUsage`: `directMemberCount` (pre-F3e count, what the satisfied interface contributes itself), `inheritedMemberCount` (distinct member ids reachable through the inheritance closure), `aggregateMemberCount` (sum), `inheritedInterfaces[]` (canonical ids, sorted ordinal so the wire shape is deterministic), `isCompositeInterface` (true iff `inheritedMemberCount > 0`). `memberCount` retains its pre-F3e name as a backwards-compatible alias for `aggregateMemberCount`. Consumer counting also walks the aggregate: a caller of an inherited member counts as a consumer of the composite facade (it reaches the contract). Pinned by `AuthorityReporterTests.AuthorityReport_CompositeInterfaceFacade_AggregatesInheritedSurface`, `AuthorityReport_NonCompositeInterface_ReportsZeroInheritedSurface`, `AuthorityReport_CompositeInterfaceConsumers_CountAcrossAggregate`, `AuthorityReport_InterfaceTypeAsHost_FindsParentInterfacesPostF3c`, and `AuthorityReport_RealGraph_CompositeHost_AggregatesInheritedMembers` (real-graph end-to-end pin via the production extractor pipeline). LB-TRACK-20260519-022 / F3e atom of the 2026-05-19 plan.

- **INV-FORWARDER-001. Method body shape is classified at extraction time, recorded on `Properties["classification"]`.** Three categories: `PureForwarder` (single-statement / expression-bodied method whose body is exactly one invocation expression), `ThinWrapper` (≤ 5 statements containing exactly one invocation overall — null-guard-then-delegate, simple-cast-then-delegate), `RealLogic` (anything else). Abstract / partial / extern methods are not classified (no entry written) — there's no body shape to categorize. Heuristic, not semantic — recorded on the type's own member, not the call target — but stable enough that an authority report can compute a coarse forwarder ratio without a second walk. Detected by `RoslynSymbolExtractor.AttachMethodClassification` (Phase P5 / LB-FR-020). Pinned by `AuthorityReporterTests.Extractor_*`.

## Execute Robustness (v0.6.7 P4)

- **INV-EXECUTE-001. `lifeblood_execute` has one reference-admission seam: `Internal.ScriptReferenceSetBuilder`.** Every execute call uses the host scripting BCL, retained workspace `CSharpCompilation` metadata references, admitted runtime probe DLLs, and the assemblies that define `RoslynSemanticView` globals. `host` is the only execution profile because scripts run in-process and share retained Roslyn workspace state; non-host `targetProfile` values are accepted for compatibility, still run against the host scripting BCL, and surface a `targetRuntimeWarnings` limitation instead of pretending to validate against a ref-pack. When the workspace is Unity-shaped (`Library/` exists at the project root), the `IRuntimeAssemblyResolver` adapter (`Lifeblood.Adapters.CSharp.UnityAssemblyResolver`) probes `Library/ScriptAssemblies/`, `Library/Bee/artifacts/`, and `Library/PackageCache/` so scripts can touch `UnityEngine` types; an empty `Library/` surfaces a `runtimeAssemblyWarnings` entry telling the caller to run a Unity build first. Runtime probe candidates pass through `System.Reflection.AssemblyName.GetAssemblyName(path)` before becoming `MetadataReference`s. Native PEs (Unity IL2CPP `GameAssembly.dll`, C++/CLI without managed surface, etc.) throw `BadImageFormatException` and are filtered at the reference-graph boundary; runtime BCL/contract assemblies (`mscorlib`, `netstandard`, `System.*`, etc.) are filtered so Unity player/stripped framework artifacts cannot shadow the host script core library; duplicate assembly identities collapse to one runtime reference; assemblies already represented by retained workspace compilations are skipped. The script-globals object (`RoslynSemanticView`) ships `Help`, `SymbolsOfKind(string)`, and `EdgesOfKind(string)` so scripts can introspect the sandbox without importing the `SymbolKind` / `EdgeKind` enums. Pinned by `ExecuteRobustnessTests` — resolver no-Library / library-without-artifacts / DLL discovery; host and non-host target-profile warning behavior; warnings propagation through success and blocked-pattern paths; native-PE and runtime-BCL filtering; runtime identity deduplication; legacy-overload delegation; `Help` content shape; string-named `SymbolsOfKind` / `EdgesOfKind` filtering. Closes LB-BUG-014, LB-BUG-007, LB-FR-012, LB-FR-013, LB-BUG-008, LB-LIM-004(a/b).

## Unity Reachability (v0.6.7 P3)

- **INV-UNITY-001. Runtime-dispatch reachability is a typed port. `IDeadCodeAnalyzer` consults `IUnityReachabilityProvider` (when injected) before flagging a symbol as dead.** The reference adapter (`Lifeblood.Adapters.CSharp.UnityReachabilityAdapter`) recognizes Unity entrypoint attributes (`RuntimeInitializeOnLoadMethod`, `InitializeOnLoadMethod`, `MenuItem`, `ContextMenu`, `PostProcessBuild`, `PostProcessScene`, `CustomEditor`, `CustomPropertyDrawer`, `Test`, `UnityTest`, …) and MonoBehaviour magic methods on types whose inheritance chain reaches a Unity message-receiver base (`UnityEngine.MonoBehaviour`, `UnityEngine.ScriptableObject`, `UnityEditor.Editor`, `UnityEditor.EditorWindow`, `UnityEngine.StateMachineBehaviour`). The adapter walks the chain via the extractor-recorded `Symbol.Properties["baseType"]` (FQN) so chains terminating in external types not loaded into the graph (UnityEngine.dll lives outside the workspace) still resolve. Cycle-safe with a 32-hop cap. Composition root injects the adapter by default; non-Unity workspaces see zero behavior change because nothing matches. Dogfooded on a real 87-module Unity workspace: dead_code findings 1095 → 729 (-33%), MonoBehaviour-magic FPs 378 → 13 (-97%). Pinned by `UnityReachabilityTests` (15 tests) covering attribute matching, magic-method detection on direct + transitive subclasses (in-graph and via external base FQN), negative cases, cycle-safety, extractor `Properties["baseType"]` + `Properties["attributes"]` recording, and end-to-end through `LifebloodDeadCodeAnalyzer`. Closes LB-FP-001, LB-FP-002, LB-BUG-009.

- **INV-UNITY-002. Asmdef edits force a full incremental re-analyze.** `AnalysisSnapshot.AsmdefTimestamps` records `*.asmdef` mtime at full-analyze time. `RoslynWorkspaceAnalyzer.IncrementalAnalyze.HasAsmdefDrift` re-scans the project root on every incremental call; any added, edited, or removed `.asmdef` triggers a full re-analyze that round. Necessary because Unity csprojs are generated from asmdefs — editing an asmdef without forcing Unity to regenerate the csproj leaves the csproj-timestamp tracker (INV-BCL-005) blind to the change, which silently runs analysis against stale module facts. Pinned by `UnityReachabilityTests` end-to-end coverage; closes the silent-stale-module class promoted from LB-NICE-003.

## Semantic Search Match Source

- **INV-SEARCH-MATCHKIND-001. `SearchResult.MatchKind` structurally reports which scoring bucket(s) drove the rank.** The reference adapter (`Lifeblood.Connectors.Mcp.LifebloodSemanticSearchProvider`) ranks symbols against three buckets — bare `Name`, `QualifiedName` (exclusive of Name per the bare-name guard), and persisted `xmlDocSummary`. Pre-fix the bucket signal lived only in the human-rendered `MatchSnippets` strings (`"name: …"` / `"qualifiedName: …"` / `"xmlDoc: …"`); callers who wanted to filter by source had to parse those strings. The first-hit-token trackers inside the adapter already knew which buckets fired — INV-SEARCH-MATCHKIND-001 lifts that to a typed `MatchKind` enum on `SearchResult`. Values: `Name`, `QualifiedName`, `XmlDoc` for single-bucket hits; `Multiple` when more than one bucket fired (e.g. token in both bare name and xmldoc summary, or qualified name and xmldoc summary). The enum is adapter-agnostic — future TypeScript / Python search adapters reuse the same taxonomy (their corpora carry the same conceptual buckets). Snippet strings are unchanged so existing renderers keep working; the structured field is purely additive. Pinned by `SemanticSearchTests.Search_*MatchKind*` (5 cases: name-only, qualifiedName-only-with-bare-name-guard, xmldoc-only, name+xmldoc Multiple, qualifiedName+xmldoc Multiple).

## Member Resolution (post-v0.7.2)

- **INV-RESOLVE-MEMBER-001. `lifeblood_resolve_member` is a type-scoped counterpart to `lifeblood_resolve_short_name`.** Where short-name resolution flattens every member matching a bare name across the entire graph (often 16+ hits on common names like `Dispose` / `Update`), `ResolveMember(graph, typeName, memberName, paramTypes?)` scopes to ONE specific containing type and returns the member surface on THAT type only. The `typeName` argument accepts canonical (`type:NS.T`), fully-qualified (`NS.T`), or bare short name (`T`); bare names dispatch through the short-name index. When the bare short name resolves to multiple types, `outcome = AmbiguousContainingType` with every candidate id returned for caller disambiguation — the resolver NEVER silently picks. The `paramTypes[]` argument filters method overloads by parameter signature (each entry is one fully-qualified parameter type); ignored for non-method members. Typed outcome enum: `Unique` / `MultipleMatches` / `NotFound` / `TypeNotFound` / `AmbiguousContainingType`. Port: `ISymbolResolver.ResolveMember` + `MemberResolutionResult`. Adapter: `LifebloodSymbolResolver.ResolveMember`. Pinned by `ResolveMemberTests` (12 cases covering every `ResolveMemberOutcome`). Closes the field-report 2026-05-11 P1 type-scoped-member ask.

## Uniform List-Shape Contract (post-v0.7.8 LB-TRACK-20260524-026/027)

- **INV-LIST-SHAPE-UNIFORM-001. Every MCP read-side tool whose response carries an unbounded list MUST expose `summarize:bool` + at least one cap argument (`maxResults` / `maxRows` / `maxTables` / `maxFiles` / `maxSites` / `maxHotspots` / `maxBoundaries` / `maxReadingOrder` / `maxMatrixEntries` / `limit`) on its input schema; descriptions reference `summarize` or `truncat*` so the catalog surfaces the wire-shape shortcut.** Eternal-shape ratchet. Pre-LB-TRACK-20260524-026/027 the list-shape contract was an emergent convention — `dead_code` / `cycles` / `blast_radius` / `context` shipped the trio early, but `file_impact` and `static_tables` lagged and overflowed downstream tool-result budgets on real-world god-types. The fix bundle landed the trio uniformly across all six list-shape tools AND added this ratchet to refuse silent drift. Future tools that ship a `list[]` response without the trio fail the ratchet at build time, not at dogfood time.
