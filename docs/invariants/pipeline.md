# Pipeline Invariants

Streaming compilation, derived file-edge truth, and incremental
re-analyze. Memory-safe by default; full retention only opted into by
the MCP server.

## Streaming Compilation Architecture (v0.3.0)

- **INV-STREAM-001**: `ModuleCompilationBuilder.ProcessInOrder` compiles one module at a time in topological order. After extraction, `Emit()` → `MetadataReference.CreateFromImage()` downgrades the full compilation (~200MB) to a lightweight PE reference (~10-100KB). Peak memory: O(1 compilation), not O(N).
- **INV-STREAM-002**: `SharedMetadataReferenceCache` deduplicates NuGet MetadataReferences across modules. One instance per `AnalyzeWorkspace` call.
- **INV-STREAM-003**: `AnalysisConfig.RetainCompilations` controls mode. `false` (default, CLI) = streaming/memory-safe. `true` (MCP server) = retained for write-side tools.
- **INV-STREAM-004**: Unity csproj support: if `<Compile Include>` items exist (old-format), use them. If absent (SDK-style), scan filesystem.
- **INV-STREAM-005**: `GraphBuilder.Build()` deduplicates ALL edges by `(sourceId, targetId, kind)`. Partial classes emit duplicate edges. The builder is the authoritative dedup boundary.

## Derived File Edges

- **INV-FILE-EDGE-001**: `GraphBuilder.Build()` derives file-level `References` edges from symbol-level edges. For each non-Contains edge between symbols in different files, a `file:X → file:Y References` edge is emitted with an `edgeCount` property. Evidence: `Inferred`, adapter: `GraphBuilder`. File edges are derived truth. Not primary.

## Incremental Re-Analyze

- **INV-INCR-001**: Incremental re-analyze (`lifeblood_analyze` with `incremental: true`) only recompiles modules whose source content changed since the last analysis. Per-file extraction results are cached in `AnalysisSnapshot`. Filesystem mtimes are a cheap prefilter only; source content hashes decide whether a touched file actually re-extracts. v1 limitation: does not cascade to dependent modules when API surface changes.

- **INV-INCREMENTAL-CONTENT-HASH-001. Incremental source-file change detection is content-authoritative, not mtime-authoritative.** `AnalysisSnapshot` stores a source content hash for every tracked source file alongside the existing extraction cache. Incremental analyze still uses mtimes to find touched candidates cheaply, but a file whose mtime changed and whose content hash is unchanged MUST return `mode:"incremental-noop"` without replacing graph slices. Responses surface both counters: `mtimeTouchedSourceFiles` counts files whose timestamp suggested work, and `contentChangedSourceFiles` counts files whose content hash actually changed. `changedSourceFiles` remains the re-extracted source-file count and therefore matches the content-changed count for normal C# source updates. This closes the mtime-only touch class where editor save/format churn invalidated otherwise stable graphs. Pinned by `IncrementalAnalyzeTests` content-hash and mtime-only no-op cases plus MCP wire-shape coverage on the counters.

- **INV-ANALYZE-AUTHORITATIVE-CHANGESET-001. `lifeblood_analyze authoritativeChangedFiles` may narrow the incremental source scan but must not suppress descriptor or scope drift checks.** When an editor, IDE, or filesystem watcher knows the exact changed-file set, the MCP edge binds it as project-relative or absolute paths and the C# adapter intersects incremental source candidates with that set before hashing. The hint is advisory for source-file scanning only: csproj/asmdef descriptor drift, module-set drift, read-only compilation-state gaps, and `excludePaths` analysis-scope drift are still evaluated independently and can still reject or widen according to `allowFullFallback`. Empty or unknown paths are normalized and ignored rather than treated as deletion authority. Pinned by `ToolArgumentContractTests` binding coverage and `IncrementalAnalyzeTests` authoritative-changed-file narrowing cases.

- **INV-INCREMENTAL-XREF-001. Cross-module edges survive incremental analyze.** After every successful `ModuleCompilationBuilder.ProcessInOrder` call, the snapshot-owned `AnalysisSnapshot.DowngradedRefs` dictionary contains exactly one entry per module the analyzer successfully compiled — including unchanged modules carried forward from prior full analyzes with their stale-but-still-valid PE metadata. The two paths agree on the contract: full analyze CLEARS `DowngradedRefs` before processing every module from scratch; incremental analyze threads the snapshot-owned carry as a SEED into `ProcessInOrder` and merges the working dict back at end-of-call. Without this, incremental analyze called `ProcessInOrder` with only the `modulesToRecompile` subset, so unchanged dependencies had no metadata reference, every cross-module symbol bound to a Roslyn error symbol, and the corresponding edges were silently dropped by `GraphBuilder`'s dangling-edge filter — a silent-data-loss class causing false-positive dead-code findings on any symbol consumed only from unchanged modules. **Acceptance criterion (now pinned):** two consecutive `analyze` calls on the same source tree — `incremental: true` then `incremental: false` — must produce identical `summary.edges`. Pinned by `IncrementalAnalyzeTests.IncrementalAnalyze_CrossModuleEdges_IdenticalAfterContentlessTouch` + `IncrementalAnalyze_CrossModuleEdges_PreservedAfterContentChangeInDependent`. Closes `LB-BUG-020` (incremental edge drop, field-report 2026-05-11).

- **INV-JSON-IMPORT-BOM-001. JSON imports accept any standard BOM, not just bare UTF-8.** `System.Text.Json.JsonSerializer.Deserialize<T>(Stream)` requires UTF-8 and throws an unhandled exception on UTF-16-LE-BOM, UTF-16-BE-BOM, UTF-8-BOM, etc. The README-documented `lifeblood export --project ... > graph.json` pattern on Windows PowerShell writes UTF-16-LE-with-BOM by default; passing the resulting file straight to `lifeblood analyze --graph graph.json` (or via the MCP `graphPath` arg) crashed pre-fix. Every Stream-to-JSON-deserialize path MUST route through `StreamReader(stream, Encoding.UTF8, detectEncodingFromByteOrderMarks: true)` and pass the resulting string to the JSON deserializer; the StreamReader auto-detects all five standard BOMs (UTF-8, UTF-8-BOM, UTF-16-LE-BOM, UTF-16-BE-BOM, UTF-32 variants) and defaults to UTF-8 when no BOM is present. Pinned by `JsonRoundTripTests.ImportDocument_AcceptsAllStandardBomEncodings` (4-case theory) plus end-to-end CLI verification via a real on-disk transcoded file. Enforcement site: `JsonGraphImporter.ImportDocument`. Other JSON-from-Stream callsites (`RulePacks.ResolveBuiltIn` reads embedded resources baked into the assembly as UTF-8 by the build, so it is not user-facing and not vulnerable to PowerShell-redirect drift; `ProcessIsolatedCodeExecutor`, `ScriptHost.Program`, and `Server.Mcp.Program` deserialize from STRINGS, not Streams, and are not vulnerable). When adding a new Stream-to-JSON-deserialize site, pair it with a BOM round-trip ratchet.

- **INV-MCP-STDIO-UTF8-001. MCP / JSON-RPC stdio ports MUST be pinned to UTF-8.** The MCP protocol mandates UTF-8 over stdio. The host process's `Console.InputEncoding` / `Console.OutputEncoding` defaults are OS-dependent — on Windows they are typically a non-UTF-8 ANSI codepage (e.g. Windows-1252, Shift-JIS), which silently mangles multi-byte characters in JSON args (Unicode identifiers, accented characters in search queries) and JSON responses. Every MCP-server composition root MUST set `Console.InputEncoding = Console.OutputEncoding = Encoding.UTF8` before any `Console.In.Read*` / `Console.Out.Write*` call. Enforcement site: `Server.Mcp.Program.Main` (top of the method, before the dispatcher loop). Defensive — no test failure has been observed, but the broader bug class is identical to `INV-JSON-IMPORT-BOM-001`: a transport that mandates UTF-8 must explicitly pin UTF-8 rather than trust the host's defaults.

- **INV-ANALYZE-STRUCTURED-FAILURE-001. The analyze pipeline never lets a raw fault escape — it wraps it with phase/module/file/profile context.** `RoslynWorkspaceAnalyzer.AnalyzeWorkspace` maintains a coarse progress cursor (phase ∈ {discovery, compilation, module-edges, graph-build}, current module, current file, current profile, reached-compilation flag) updated at every phase boundary, inside the `onModuleProgress` callback, and per syntax tree. Any UNEXPECTED exception is caught at the method boundary and re-thrown as a `WorkspaceAnalysisException` carrying that cursor + the inner exception; an already-structured `WorkspaceAnalysisException` re-throws unchanged, and deliberate caller-input validation (`ArgumentException`, e.g. an unknown define-profile name) propagates as its typed contract rather than being masked as a pipeline fault. The MCP handler (`ToolHandler.HandleAnalyze`) catches the typed exception and serializes `{phase, module, filePath, profile, failedBeforeCompilation, exceptionType, message}` instead of letting the dispatcher emit a bare `"Error: Object reference not set to an instance of an object."`. Motivation: agents use full analyze as the trust reset; an opaque collapse there (observed as a NullReference after Unity asset-import churn, `LB-INBOX-012`) forces a blind fall back to grep/IDE with no idea which phase is unsafe. The exception type is language-agnostic so any `IWorkspaceAnalyzer` reuses it. Pinned by `WorkspaceAnalysisFailureTests.AnalyzeWorkspace_PipelineFault_ThrowsStructuredException_WithPhaseContext`.

- **INV-ANALYZE-FALLBACK-001. Scope-widening is a caller policy, not an adapter decision.** When the adapter detects drift it cannot honor cheaply (no prior cache, module set changed since the snapshot, project descriptor edited, analysis-scope options changed, or retained compilation state unavailable after a read-only analyze), the response shape is determined by the caller's `AnalysisConfig.AllowFullFallback` flag. **Default `false`** (fail-loud): the adapter returns `IncrementalAnalyzeResult { Mode = Rejected, Graph = null, Reason = ..., Detail = ... }` - no work done, caller decides next step. **Opt-in `true`**: the adapter widens to a full re-analyze and returns `Mode = FullFallback, Graph = ..., Reason = ...` so the result lands AND the cache miss stays visible. The reason taxonomy is adapter-agnostic - `NoPriorAnalysis`, `ModuleSetChanged`, `ModuleDescriptorChanged`, `AnalysisScopeChanged`, `CompilationStateUnavailable` - so future Python / TypeScript / etc. adapters reuse the same shape; the adapter-specific descriptor kind (`asmdef`, `csproj`, `pyproject.toml`), scope kind (`excludePaths`), or retained-state detail lives in `Detail`. Internal best-effort callers like `GraphSession.MaybeRefreshIfStale` deliberately set `AllowFullFallback = true` because their contract is "make state fresh"; user-facing callers route the user's own flag through unchanged. **NoPriorAnalysis is always Rejected regardless of flag** - without a snapshot there is no `projectRoot` to widen against; remediation is fixed (call `AnalyzeWorkspace` first). Wire shape on the MCP layer: `mode` reports what the adapter DID (`full` / `incremental` / `incremental-noop` / `rejected`); `requestedMode` separately reports what the caller ASKED (`full` / `incremental`); `fallbackReason` + `fallbackDetail` populate alongside whenever the cheap path could not be honored cleanly; rejection responses additionally carry `canRetryFull: true` and a `suggestedRetry: { incremental: true, allowFullFallback: true }` block so the agent's next move is self-documenting and requires no out-of-band knowledge. Rejection is a NORMAL structured result, not a transport / tool error. Pinned by `IncrementalAnalyzeTests` (adapter-level cases including NoPriorAnalysis-always-rejects + ModuleSetChanged x {Rejected, FullFallback}) and `AnalyzeWireShapeTests` (wire-level ratchets covering `full` / `incremental` / `incremental-noop` / `rejected` / fall-back-to-full shapes, first-call-incremental rejects, `analysisScopeChanged` when `excludePaths` changes, and `compilationStateUnavailable` recovery after read-only analysis).

- **INV-READONLY-RECOVERY-001. A read-only MCP analyze must not leave the session in a crash-prone half-retained state.** `readOnly:true` intentionally streams and releases Roslyn compilations so write-side tools are unavailable. If the next call requests incremental retained analysis without a retained compilation snapshot, `lifeblood_analyze` MUST return a structured `mode:"rejected"` response with `fallbackReason:"compilationStateUnavailable"`, `canRetryFull:true`, and the same `suggestedRetry` shape as other fallback cases. With `allowFullFallback:true`, the server MUST run a full retained analyze and rebuild the `RoslynSemanticView` instead of trying to incrementally reuse missing compilation state. Pinned by `AnalyzeWireShapeTests` and direct stdio smoke coverage against the rebuilt MCP server.

- **INV-ANALYZE-EXCLUDEPATHS-001. `lifeblood_analyze excludePaths` removes files from analysis scope before Roslyn compilation and changing that set is incremental scope drift.** The MCP edge accepts project-relative POSIX path globs with the shared `PathGlobMatcher` grammar (`*` = any run including `/`, `?` = one character, full-path case-insensitive match). `ModuleCompilationBuilder` applies the compiled glob set before parsing syntax trees, so excluded files never enter Roslyn, extraction, graph validation, diagnostics, or retained compilations. `AnalysisSnapshot` records the normalized exclude set; incremental analyze compares the requested set to the retained set and returns `FallbackReason.AnalysisScopeChanged` / wire `fallbackReason:"analysisScopeChanged"` unless `allowFullFallback:true` permits a full re-analyze. `GraphSession` remembers the active exclude set for stale auto-refresh and clears it on JSON graph import. Pinned by `AnalyzeWireShapeTests.Load_FullAnalyze_ExcludePaths_DropsMatchingSourceFromGraph`, `Load_IncrementalWithChangedExcludePaths_RejectsWithAnalysisScopeChanged`, `CsprojCompilationFactsTests.Compilation_ExcludePathGlobs_DropsMatchingSourceBeforeGraphExtraction`, `ToolArgumentContractTests.BindAnalyze_ExcludePaths_*`, and the `lifeblood_analyze` schema snapshot. Closes the analyze-scope half of `LB-INTAKE-20260601-004`.
