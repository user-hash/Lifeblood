# Architecture Invariants

Hexagonal core: pure Domain, ports-only Application, isolated adapters
and connectors. Every invariant on this page is enforced by ratchet
tests under `tests/Lifeblood.Tests/ArchitectureInvariantTests.cs`.

## Domain Purity

- **INV-DOMAIN-001**: `Lifeblood.Domain` has ZERO dependencies. Not Roslyn, not JSON, not System.IO, not anything. If a PackageReference appears, the architecture is broken.
- **INV-DOMAIN-002**: Domain never references Application, Adapters, Connectors, or CLI.

## Application Layer

- **INV-APP-001**: Application depends only on Domain.
- **INV-APP-002**: Application never references concrete adapters or connectors. Only port interfaces.

## Graph Model

- **INV-GRAPH-001**: SymbolKind enum is language-agnostic. No C#-isms, no Python-isms.
- **INV-GRAPH-002**: Language-specific metadata goes in `Symbol.Properties` dictionary.
- **INV-GRAPH-003**: Every edge carries Evidence (kind, adapter, confidence, source span).
- **INV-GRAPH-004**: Analyzers do NOT modify the graph. Results are separate objects. The graph is read-only after construction.

## Left Side (Language Adapters)

- **INV-ADAPT-001**: Every adapter declares capabilities honestly via AdapterCapability.
- **INV-ADAPT-002**: C# adapter is the reference. Most complete, best tested.
- **INV-ADAPT-003**: External adapters communicate via JSON graph schema only.
- **INV-ADAPT-004**: No adapter code leaks into Domain or Application.
- **INV-NATIVE-CLANG-BOUNDARY-001. Native Clang is an external JSON adapter, not a managed Lifeblood module.** `adapters/native-clang` builds a standalone executable that emits `schemas/graph.schema.json`; Lifeblood imports the graph through `JsonGraphImporter`. Native source/build files must not reference `Lifeblood.Domain`, `Lifeblood.Application`, `Lifeblood.Analysis`, managed adapters, connectors, CLI, or MCP server projects. Ratchet-tested by `NativeClangArchitectureTests.NativeClang_SourceAndBuildFilesDoNotReferenceManagedLifebloodModules`.
- **INV-NATIVE-CLANG-LIBCLANG-001. Raw libclang API usage has an explicit leak budget.** During the beta-to-reference-architecture transition, every file that touches `clang-c`, `CX*`, or `clang_*` is listed in `NativeClangArchitectureTests.NativeClang_LibClangApiUsageStaysInsideExplicitBoundary`. New raw libclang touch points require an intentional allowlist edit. The shrink path is to translate libclang cursors into native fact DTOs at the adapter edge; function, global, and type-member symbol emission are now graph-side leaves behind declaration facts.
- **INV-NATIVE-CLANG-RATCHET-001. Native executable ratchets can be fail-hard on demand.** `NativeClangExecutableRatchetTests` use explicit skip semantics when the native executable is absent, but `LIFEBLOOD_REQUIRE_NATIVE_CLANG=1` turns absence into a hard failure. Native release validation should set this variable so the suite cannot pass by skipping executable coverage.

## Right Side (AI Connectors)

- **INV-CONN-001**: Connectors depend on Application ports, not on adapters.
- **INV-CONN-002**: MCP connector serves the graph read-only.
- **INV-CONN-003**: Context pack generator produces AI-consumable JSON, not human prose.

## Analysis

- **INV-ANALYSIS-001**: All analyzers are stateless. Input: graph + config. Output: typed result.
- **INV-ANALYSIS-002**: No analyzer modifies the graph. Read-only.
- **INV-ANALYSIS-003**: CouplingAnalyzer counts distinct dependants, not edge count.

## Testing

- **INV-TEST-001**: Every adapter passes the same golden repo contract tests.
- **INV-TEST-002**: Every analyzer is tested against golden repos.

## Pipeline

- **INV-PIPE-001**: The pipeline is deterministic. Same input = same output.

## Process Isolation

- **INV-SCRIPTHOST-001. `Lifeblood.ScriptHost` has zero `ProjectReference`.** The script host is a process-isolated child that runs untrusted code; its "no access to parent state" guarantee depends on not taking a ProjectReference to any Lifeblood module. NuGet PackageReferences are allowed (Microsoft.CodeAnalysis.CSharp.Scripting is load-bearing). Ratchet-tested by `ArchitectureInvariantTests.ScriptHost_HasZeroProjectReferences`.

## Multi-Define Profile Resolution (post-v0.7.8 Wave 6.A, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-RESOLVER-001. `IDefineProfileResolver` is the single Application-layer seam that returns the set of preprocessor-symbol profiles a project should be analyzed under; the default adapter `DefaultDefineProfileResolver` returns exactly one identity Editor profile so every existing single-profile callsite keeps its byte-stable wire shape.** Wave 6.A of the L-LIM-001 multi-define union analyze rollout. Port lives in `Lifeblood.Application.Ports.Left.IDefineProfileResolver`; DTO is `DefineProfile { Name, AddDefines[], RemoveDefines[] }`. Per-profile active define set is computed as `(BASE - RemoveDefines) ∪ AddDefines` where BASE is the csproj `DefineConstants` already extracted by `RoslynModuleDiscovery`; the operation is ordinal-sorted for byte-stable provenance. The Editor identity profile carries empty `AddDefines` and `RemoveDefines` so the active set equals BASE — analyzer behavior is unchanged when the default resolver is in use. Adapter sibling `UnityDefineProfileResolver` (Wave 6.C) adds the canonical 2-profile MVP (Editor + Player); pluggable resolvers (Xamarin, multi-TFM, custom workspaces) ship as sibling adapters without touching the analyzer. Pinned by `DefineProfileResolverTests` (4 facts: single Editor profile, identity Add/Remove empty, idempotent across calls, DefineProfile immutability). LB-TRACK-L-LIM-001 / Wave 6.A.

## Multi-Define Compile Pipeline (post-v0.7.8 Wave 6.B, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-ANALYZE-001. `AnalysisConfig.DefineProfiles` is the single seam that switches `RoslynWorkspaceAnalyzer` from single-profile back-compat into multi-profile union analyze; the orchestrator runs N sequential per-profile passes, attributes each pass's edges with the profile name, and unions edge profile sets at the GraphBuilder dedup seam.** Wave 6.B of L-LIM-001 multi-define union analyze rollout. Single-profile back-compat (`DefineProfiles == null` or empty) executes ONE pass against the resolver's first profile (Default = identity Editor); edges keep `Profiles = null`; wire shape byte-stable with pre-Wave-6 behavior. Multi-profile (`DefineProfiles = ["Editor", "Player"]` etc.) runs one pass per requested profile; per-pass module clones replace `PreprocessorSymbols` via `DefineProfileApplier.WithProfileDefines`; per-pass edges are tagged via `EdgeProfileTagger.Tag(rawEdges, profileName)` and appended to the snapshot through `AnalysisSnapshot.AppendProfileEdges` (first pass uses `ReplaceFile` to set baseline symbols + file timestamps; subsequent passes only add profile-tagged edges). Unknown profile names throw `ArgumentException` so the caller sees the typo eagerly — silent fallback is forbidden. `IDefineProfileResolver.ResolveProfiles(projectRoot)` is invoked once per analyze; the returned vocabulary is the resolver's authority, never overridden by the analyzer. Symbol set across profiles is union-by-id (existing GraphBuilder dedup); future Wave 6.B+ atom would tag symbols too if profile-discriminated symbol presence becomes load-bearing. Pinned by `MultiProfileAnalyzeTests` (11 facts: applier add/remove ordinal-sort + dedup, identity profile preserves baseline, tagger null/named, merger union/identity/double-null, GraphBuilder union dedup, single-profile back-compat keeps Profiles=null, AnalysisConfig.DefineProfiles defaults null, unknown profile name throws).

- **INV-MULTI-DEFINE-EDGE-PROFILES-001. `Edge.Profiles` is the per-edge provenance field that records which define profiles observed the edge during analyze; null on single-profile back-compat, populated set on multi-profile, empty array on multi-profile is a bug.** GraphBuilder dedup keeps first-write-wins for every Edge field except Profiles: when the same `(SourceId, TargetId, Kind)` is observed under multiple profiles, `EdgeProfileMerger.MergeProfiles` unions the two `Profiles` sets into an ordinal-sorted distinct list and stamps the clone over the existing entry. Single-profile flow: every observed edge has `Profiles = null`, merge is no-op, wire shape byte-stable. Multi-profile flow: the orchestrator's per-profile `EdgeProfileTagger.Tag` pass injects `Profiles = [activeProfileName]` on every edge before snapshot append; the dedup pass unions them. JSON graph round-trip via `JsonGraphExporter` / `JsonGraphImporter` preserves the field. Pinned by `MultiProfileAnalyzeTests` GraphBuilder union dedup case + back-compat case.

- **INV-MULTI-DEFINE-APPLIER-001. `DefineProfileApplier.Apply(baseSymbols, profile)` is the single pure function that derives the active preprocessor-symbol set for a profile pass; result is ordinal-sorted + distinct.** `(BASE - profile.RemoveDefines) ∪ profile.AddDefines`, normalized via `OrderBy + Distinct`. `WithProfileDefines(module, profile)` returns a shallow `ModuleInfo` clone with `PreprocessorSymbols` replaced — every other csproj-discovered field stays identity-equal to the input (BclOwnership, ReferenceClosure, LangVersion, NullableContext, NoWarn, IVT, dependency list, file list, properties). The applier never mutates the input ModuleInfo; the orchestrator's per-profile pass builds a fresh ModuleInfo array each iteration. Pinned by `MultiProfileAnalyzeTests.DefineProfileApplier_*` (2 facts: add+remove ordinal-sort dedup, identity profile preserves baseline byte-stable).

## Unity Multi-Define Resolver (post-v0.7.8 Wave 6.C, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-UNITY-RESOLVER-001. `UnityDefineProfileResolver` returns the 2-profile MVP (`Editor` identity + `Player` minus the Unity Editor discriminator family) on Unity workspaces; falls back to single Editor identity on non-Unity workspaces.** Workspace detection: `Library/` directory exists at project root. The Player profile's `RemoveDefines` set is the canonical Unity Editor discriminator vocabulary: `UNITY_EDITOR`, `UNITY_EDITOR_WIN`, `UNITY_EDITOR_64`, `UNITY_EDITOR_OSX`, `UNITY_EDITOR_LINUX`. Player adds NOTHING — platform-target defines (`UNITY_ANDROID`, `UNITY_IOS`, `UNITY_WEBGL`, `PLATFORM_*`, `ENABLE_IL2CPP`) are ALREADY in the baseline csproj `DefineConstants` on a workspace whose Editor targets that platform. L-LIM-001's load-bearing root cause is the `UNITY_EDITOR` axis — every `#if (UNITY_ANDROID || UNITY_IOS) && !UNITY_EDITOR` canonical witness flips from inactive to active on the Player profile. The 5-platform variant (Android / iOS / WebGL / Standalone as distinct profiles) is a v2 enhancement that requires reading Unity's `PlayerSettings.GetScriptingDefineSymbolsForGroup` — out of Wave 6 scope. Eternal vocabulary extension: any new `UNITY_EDITOR*` symbol Unity ships must extend `UnityEditorDiscriminators`. Non-Unity workspace fallback returns single-Editor-identity-only so the resolver is safe to inject into every composition root regardless of workspace flavor. Pinned by `UnityDefineProfileResolverTests` (8 facts: 2-profile shape, Editor identity, Player removes 5 editor discriminators, non-Unity fallback, Player applied to baseline strips editor discriminators, Player no-op when baseline has none, idempotent repeated calls, asmdef-vs-SDK-style csproj treated identically — vocabulary is workspace-shape-aware not csproj-flavor-aware).

## Multi-Define Wire Shape (post-v0.7.8 Wave 6.D, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-WIRE-001. `lifeblood_analyze` accepts `defineProfiles:string[]?` input; response carries `profileCount`, `activeProfiles`, and `perProfileEdgeCounts` on the summary. `lifeblood_dependants` / `lifeblood_dependencies` surface `profiles[]` per edge wire-side and accept an optional `profileFilter:string[]?` arg that narrows results to edges whose Profiles intersect the filter; edges with Profiles=null (single-profile back-compat) pass every filter.** Wave 6.D wire surface for L-LIM-001 multi-define. `EdgeDetail.Profiles` plumbs `Edge.Profiles` from the graph through the `IMcpGraphProvider` port to `BuildEdgeWire` in the handler. `ApplyProfileFilter` is a pure post-query narrow that walks the EdgeDetail array; null `profileFilter` short-circuits to identity. `GraphSession.Load` accepts the optional profile array and threads it through `AnalysisConfig.DefineProfiles` to the analyzer. `GraphSession` injects `UnityDefineProfileResolver` by default — auto-detects `Library/` and returns 2 profiles on Unity, single Editor identity on non-Unity, so the injection is safe at every composition root regardless of workspace flavor. Pinned by `MultiProfileWireShapeTests` (6 facts: dependants surfaces profiles, dependencies surfaces profiles, profileFilter narrows to matching profile, profileFilter unmatched set returns empty, no filter keeps all edges, single-profile-null edges pass filter as back-compat).

## Multi-Define Incremental Parity (post-v0.7.8 reviewer Stage 1, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-INCREMENTAL-001. `AnalysisSnapshot.ActiveProfiles` is the single source of truth for which profiles a graph is under; `RoslynWorkspaceAnalyzer.IncrementalAnalyze` MUST replay that profile set over changed files so per-edge `Profiles[]` provenance survives a file-touch.** The snapshot retains `ActiveProfiles` (set at full-analyze time); the analyzer exposes `RetainedProfileNames` publicly. `IncrementalAnalyze` runs one pass per snapshot profile over the changed-module set — `isFirstProfile` retains compilations + owns skipped-files + downgraded-refs (`INV-MULTI-DEFINE-IOP-001` + `INV-INCREMENTAL-XREF-001` carry through), subsequent passes force streaming mode and `AppendProfileEdges`. Single-profile snapshots (Count == 1) collapse to a single null-tagged pass — wire shape byte-stable with pre-multi-define behavior. `GraphSession.LoadIncremental` echoes `RetainedProfileNames` on the wire response when Count >= 2. Pinned by `MultiProfileIncrementalParityTests` (multi-profile preserves Player-only edge across file-touch + profileFilter parity; single-profile keeps `Edge.Profiles=null` byte-stable).

## IOperation Tool Multi-Profile Scope (post-v0.7.8 Wave 6.E, LB-TRACK-L-LIM-001)

- **INV-MULTI-DEFINE-IOP-001. `lifeblood_enum_coverage`, `lifeblood_static_tables`, `lifeblood_assignment_coverage` operate against the retained profile's compilations only; `RoslynWorkspaceAnalyzer.RetainedProfileName` records which profile that is (= first profile in `AnalysisConfig.DefineProfiles`, or the resolver's first profile in single-profile back-compat). Responses surface `analyzedUnderProfile`; the optional `profileScope` input is honored when it matches the retained name and fails loudly with a switch-instructions error otherwise.** Memory architecture: first profile pass uses caller-passed `RetainCompilations`; every subsequent profile pass forces `RetainCompilations=false` so compilations downgrade immediately after edge extraction — peak RAM stays at single-profile baseline regardless of profile count. Caller controls which profile is retained by ordering their `defineProfiles` list (Editor first → IOperation tools see Editor; Player first → IOperation tools see Player; re-analyze to switch). Cross-profile IOperation comparison ("how does this enum's coverage differ under Editor vs Player?") is v2 — requires multi-profile compilation retention which doubles RAM. The single-retained-profile MVP solves the immediate L-LIM-001 trap (an agent running `enum_coverage` under Editor never sees the player-only branches) by being explicit about what's covered: `analyzedUnderProfile` answers "which profile are these counts from?" on every response. Pinned by `MultiProfileIOpScopeTests` (4 facts: RetainedProfileName null before analyze, equals first profile under single-profile, equals first under multi-profile, respects caller order Player-first).

- **INV-MULTI-DEFINE-WRITESIDE-001. `lifeblood_find_references`, `lifeblood_find_definition`, `lifeblood_find_implementations`, `lifeblood_rename` walk the retained profile's compilations only. Multi-profile snapshots SHIP an inline `limitations[]` entry naming the retained profile + the other-profile gap on EVERY response; `analyzedUnderProfile` is always emitted. Optional `profileScope` matches the retained-profile honesty gate from `INV-MULTI-DEFINE-IOP-001` — mismatched values fail loudly with switch-instructions, never return the wrong profile's view silently.** Closes the contract gap external review 2026-05-24 surfaced: prior to this invariant, the CHANGELOG Wave 6 claim "L-LIM-001 closed end-to-end for `find_references` / `dependants`" was overbroad because `dependants` got `profileFilter` over the union graph while `find_references` continued to call the retained `ICompilationHost` directly with no profile awareness. A user analyzing under `defineProfiles:["Editor","Player"]` and then running `find_references` would silently miss `#if !UNITY_EDITOR` callsites. Fix discipline: every write-side Roslyn tool now surfaces `analyzedUnderProfile` + (on multi-profile snapshots) a `limitations[]` entry naming the retained profile + the profiles whose preprocessor-guarded sites are NOT in the response, alongside switch-instructions ("re-analyze with the target profile first in `defineProfiles`"). Cross-profile retention (which would let `find_references` query Player without re-analyze) is deferred — multiplies peak RAM per retained profile. For union-graph reference queries that span profiles, route through `lifeblood_dependants` / `lifeblood_dependencies` with `profileFilter`. Pinned by `MultiProfileWriteSideScopeTests` (7 facts: analyzedUnderProfile + limitations surface on find_references/find_definition/find_implementations/rename under multi-profile; profileScope mismatch errors loudly; profileScope match succeeds; single-profile graphs emit empty limitations[]).

## Composition-Root Allowlist

- **INV-COMPROOT-001. Composition roots use only the allowlist.** `Lifeblood.CLI` and `Lifeblood.Server.Mcp` reference only `{Domain, Application, Analysis, Adapters.CSharp, Adapters.JsonGraph, Connectors.ContextPack, Connectors.Mcp}`. The allowlist is declared once as `private static readonly HashSet<string> CompositionRootAllowedModules` on `ArchitectureInvariantTests`. Single source of truth. Expanding it is a conscious architectural decision requiring a commit that edits the test. Ratchet-tested by `CompositionRoot_CLI_UsesOnlyAllowedModules` and `CompositionRoot_ServerMcp_UsesOnlyAllowedModules`.
