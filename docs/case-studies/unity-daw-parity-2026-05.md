# Case study: bringing Lifeblood diagnose to parity with `dotnet build` on a 90-module Unity workspace

**Workspace shape:** 90-module Unity / .NET project (anonymized as
"Project A"). Mix of SDK-style csprojs for tooling, MSBuild 2003-schema
csprojs auto-generated by Unity's asmdef pipeline. ~62k symbols / ~225k
graph edges at full analyze, ~120 cycles after taxonomy filtering.
xunit + Microsoft.NET.Test.Sdk test assemblies that cross
`InternalsVisibleTo` boundaries into the production code.

**Lifeblood version under test:** v0.7.5 → unreleased
(`Wave W1-A` … `Wave W3-A`, v0.7.6 prep).

## The friction

Project A's CI was green: `dotnet build` produced zero warnings and
zero errors across all 90 modules. Yet running `lifeblood_diagnose
moduleName=Project.Tests` against the same workspace returned **7,772
diagnostics**:

| Diagnostic | Count | Class |
|-----------|------:|-------|
| `CS1701` | 7,537 | Cross-module type-ref binding-redirect mismatch |
| `CS0122` |   223 | Inaccessible due to protection level |
| `CS1503` |     6 | Cannot convert argument type |
| `CS0117` |     5 | Type does not contain member |
| `CS1729` |     1 | Type does not contain matching constructor |

Every CS0122 was a test method touching production code marked
`internal` — exactly what `<InternalsVisibleTo>` was supposed to grant.
Every CS1701 was an upstream PE (xunit.core) carrying a TypeRef to
`System.Runtime 4.0.0.0` while Lifeblood compiled against the BCL ref
pack at `8.0.0.0`.

For the agent driving Lifeblood, the result was unusable: every "is
this code broken?" question required a manual cross-check against
`dotnet build` before action. The trust budget for "verified by
Lifeblood, ship it" had collapsed to zero on this workspace shape.

## The diagnosis path

The findings were two distinct classes pretending to be one:

1. **CS1701 family** is a cross-module TypeRef binding-redirect
   warning Roslyn emits per consuming type-ref. MSBuild silently
   suppresses both `CS1701` and `CS1702` via the documented
   `Microsoft.CSharp.CurrentVersion.targets` baseline
   (`<NoWarn>$(NoWarn);1701;1702</NoWarn>`) on every csc invocation.
   Lifeblood compiled without that baseline.

2. **CS0122 family** was a separate root cause: Project.Tests's
   production-side adapter project declared `<InternalsVisibleTo
   Include="Project.Tests" />` in the csproj, and MSBuild's
   `GenerateAssemblyInfo` target writes the corresponding
   `[assembly: InternalsVisibleTo("Project.Tests")]` attribute into
   `obj/<Tfm>/<AssemblyName>.AssemblyInfo.cs`. Lifeblood's SDK-style
   source scan correctly skips `obj/`, so the attribute never reached
   the compilation. The producer PE Lifeblood emitted carried no
   friend-assembly metadata and every internal access from the test
   module fired CS0122.

Investigation tools: Lifeblood `lifeblood_diagnose` itself surfaced
the message text, the user reported the discrepancy against
`dotnet build` on the same compilation, and the breakdown into
diagnostic IDs let the gap split cleanly into the two product classes.

## The fix

Shipped in four atomic commits across the v0.7.6 prep wave:

| Commit | INV |
|--------|-----|
| `f0939ff` | `INV-DIAGNOSTIC-NUGET-BINDING-PARITY-001` — top-level reference-set unification by simple-name, highest version wins (mirrors MSBuild `<AutoUnify>true</AutoUnify>`) |
| `f0b56a0` | `INV-DIAGNOSTIC-MSBUILD-IMPLICIT-NOWARN-001` — `CS1701` / `CS1702` baseline NoWarn unioned into every discovered module (mirrors `Microsoft.CSharp.CurrentVersion.targets`) |
| `fea89f9` | `INV-DIAGNOSTIC-IVT-PARITY-001` — `<InternalsVisibleTo>` items synthesize an `[assembly: InternalsVisibleTo("X")]` syntax tree at compilation time |
| `7baee98` | `INV-DIAGNOSTIC-PARITY-001` — ratchet wall: Lifeblood-self diagnose never fires the canonical parity diagnostic ID set |

The wall ratchet (`BuildDiagnosticParityTests`) runs the full discovery
+ compilation builder pipeline against Lifeblood's own source tree on
every test invocation and asserts an empty parity-class diagnostic
subset. Lifeblood self is the dogfood fixture: every commit on `main`
must pass `dotnet build` before reaching the test suite, so any
parity-class diagnostic firing under the pipeline is by definition a
Lifeblood-side false positive.

## The outcome

Same workspace, same compilation, after the wave:

| Diagnostic | Pre-W1 | Post-W1 |
|-----------|------:|------:|
| `CS1701` | 7,537 | 0 |
| `CS0122` |   223 | 0 |
| `CS1503` |     6 | 0 |
| `CS0117` |     5 | 0 |
| `CS1729` |     1 | 0 |
| **Total** | **7,772** | **0** |

Lifeblood `diagnose` now returns the same diagnostic set `dotnet build`
returns. The trust budget the workspace owner spent on manual
cross-checks dropped to zero. `lifeblood_compile_check` is usable as a
pre-commit gate without the per-message escape hatch. The ratchet wall
prevents the regression in future releases.

## What's deliberately not claimed

- The dedup helper (W1-A) alone did NOT close CS1701. The empirical
  post-dedup count was 7,642 — almost identical to pre-dedup. The
  cross-module TypeRef class is structurally unreachable from
  top-level reference deduplication. The complete fix required
  mirroring MSBuild's implicit NoWarn baseline (W1-Ab) on top. This
  finding is documented in the `INV-DIAGNOSTIC-NUGET-BINDING-PARITY-001`
  "Scope note" so a future audit does not re-propose dedup as a
  one-shot fix.
- Lifeblood does not invoke MSBuild. The implicit-NoWarn baseline is a
  fixed list (`["CS1701", "CS1702"]`) sourced from the documented
  contract in `Microsoft.CSharp.CurrentVersion.targets`. If a future
  MSBuild SDK extends the baseline, Lifeblood will need a follow-up
  commit; the list is one place in `RoslynModuleDiscovery` so the
  drift is one line away when it happens.
- Project A's name and module names are anonymized. The empirical
  counts and the gap shape come from real measurements on the live
  workspace.

## Reproducing the wall

```bash
cd Lifeblood
dotnet test Lifeblood.sln -c Debug --filter "FullyQualifiedName~BuildDiagnosticParityTests"
```

Expected output: `Passed: 1, Failed: 0`. Any failure surfaces a
per-module-per-ID breakdown with sample message text so the regression
points at its INV directly.
